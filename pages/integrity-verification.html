Introduction

The integrity and authenticity of canonical documents are essential to the No Fate framework. Users must be able to verify independently that documents have not been altered and that they originate from authorized stewards.

This page explains how to verify canonical documents using cryptographic methods:

  • SHA-256 hashing verifies that a document has not been modified (integrity).
  • GPG signatures verify that a document was signed by an authorized steward (authenticity).

Both methods are industry-standard cryptographic techniques. They do not require trust in third partiesâ€”verification is performed entirely on your local system.

What "Canonical" Means

A canonical document is an official, authoritative version of a No Fate specification. Canonical documents are immutable once published. They serve as stable reference points for implementers, auditors, and compliance assessments.

Characteristics of Canonical Documents

  • Immutable: Once published, they cannot be modified. Changes require publishing a new versioned document.
  • Versioned: Each canonical document has a semantic version number (e.g., v1.0.0).
  • Cryptographically Verified: Each document includes a SHA-256 hash and may include a GPG signature.
  • Publicly Accessible: Canonical documents are freely available for download and verification.

Why Verification Matters

Verification ensures that:

  • The document you downloaded is identical to the published canonical version.
  • No alterations occurred during transmission or storage.
  • The document originates from authorized stewards, not a malicious third party.

Without verification, you cannot be certain that a document is authentic or unaltered.

SHA-256 Hash Verification

SHA-256 is a cryptographic hash function that produces a unique 64-character fingerprint for any file. If even a single character changes, the hash will be completely different.

How SHA-256 Verification Works

  1. The steward computes the SHA-256 hash of the canonical document and publishes it alongside the document.
  2. You download the document and compute its SHA-256 hash on your local system.
  3. You compare your computed hash with the published hash. If they match, the document is authentic and unaltered.

Step-by-Step Instructions

Step 1: Obtain the Published Hash

Find the published SHA-256 hash for the canonical document. This is typically provided in:

  • The document's metadata section
  • A checksums.txt file in the repository
  • Release notes or documentation pages

Example published hash:

abc123def456789abcdef0123456789abcdef0123456789abcdef0123456789  No_Fate_Contract_v1_0_0.md

Step 2: Download the Canonical Document

Download the canonical document from the official repository or website. Ensure you are downloading from a trusted source.

Step 3: Compute the SHA-256 Hash

Use a command-line tool to compute the hash of the downloaded file.

Using sha256sum:

sha256sum No_Fate_Contract_v1_0_0.md

Using shasum:

shasum -a 256 No_Fate_Contract_v1_0_0.md

Using PowerShell:

Get-FileHash -Algorithm SHA256 No_Fate_Contract_v1_0_0.md

The command will output a 64-character hexadecimal string.

Step 4: Compare the Hashes

Compare the hash you computed with the published hash:

  • If the hashes match: The document is authentic and unaltered. You can trust its contents.
  • If the hashes do not match: The document has been modified or corrupted. Do not trust it. Download a fresh copy from the official source and verify again.

Example Verification

Suppose the published hash is:

abc123def456789abcdef0123456789abcdef0123456789abcdef0123456789

You compute the hash of your downloaded file:

$ sha256sum No_Fate_Contract_v1_0_0.md
abc123def456789abcdef0123456789abcdef0123456789abcdef0123456789  No_Fate_Contract_v1_0_0.md

The hashes match. The document is verified.

Optional GPG Signature Verification

GPG (GNU Privacy Guard) signatures provide an additional layer of verification by confirming that a document was signed by an authorized steward's private key.

How GPG Verification Works

  1. The steward signs the canonical document using their private GPG key, generating a .sig signature file.
  2. The steward publishes their public GPG key, which anyone can obtain.
  3. You download the document, the signature file, and the public key.
  4. You use GPG to verify that the signature matches the document and was created by the steward's private key.

Why Use GPG Signatures?

GPG signatures verify authenticity in addition to integrity:

  • SHA-256 verifies the document has not been altered.
  • GPG verifies the document was signed by a specific authorized steward.

If a malicious actor creates a fake document and publishes a fake hash, GPG verification will fail because they do not possess the steward's private key.

High-Level GPG Verification Steps

Step 1: Obtain the Steward's Public Key

Download the steward's public GPG key from the official repository or keyserver. Example key ID: <KEYID>.

gpg --recv-keys <KEYID>

Step 2: Download the Signature File

Signature files typically have a .sig extension. Example:

No_Fate_Contract_v1_0_0.md.sig

Step 3: Verify the Signature

Use GPG to verify the signature against the document:

gpg --verify No_Fate_Contract_v1_0_0.md.sig No_Fate_Contract_v1_0_0.md

If verification succeeds, GPG will output a message indicating a good signature from the steward's key:

gpg: Good signature from "No Fate Steward <email@example.org>" [ultimate]

If verification fails, GPG will indicate a bad signature or that the signature does not match the document.

Trust and Key Management

GPG verification depends on obtaining the correct public key. Ensure you download the key from:

  • The official No Fate repository
  • Trusted keyservers
  • Multiple independent sources (to cross-verify)

Be cautious of keys obtained from unverified sources.

Checksums Files

A checksums file (often named checksums.txt or SHA256SUMS) contains SHA-256 hashes for multiple canonical documents in a single file. This simplifies verification when multiple documents are published together.

Format

A typical checksums file contains one line per document:

abc123def456...  No_Fate_Contract_v1_0_0.md
def456abc789...  Deterministic_Map_of_Law_v1_0_0.md
789abc123def...  AI_Refusal_Specification_v1_0_0.md

Each line contains:

  • The 64-character SHA-256 hash
  • Two spaces
  • The filename

Using Checksums Files

Verify All Documents at Once

If you have downloaded multiple canonical documents, you can verify all of them using the checksums file:

sha256sum -c checksums.txt

This command checks each file listed in the checksums file and reports whether each hash matches:

No_Fate_Contract_v1_0_0.md: OK
Deterministic_Map_of_Law_v1_0_0.md: OK
AI_Refusal_Specification_v1_0_0.md: OK

Verify a Single Document

If you only want to verify one document, you can extract its hash from the checksums file and compare manually, or use:

grep No_Fate_Contract_v1_0_0.md checksums.txt | sha256sum -c

Verifying the Checksums File Itself

The checksums file may itself be GPG-signed to ensure its authenticity. If a checksums.txt.sig file is provided, verify it using GPG:

gpg --verify checksums.txt.sig checksums.txt

This confirms that the checksums file was created by an authorized steward.

Integrity Guarantees and Limitations

What Verification Guarantees

  • Integrity: SHA-256 verification guarantees that the document has not been altered since the hash was computed.
  • Authenticity: GPG verification guarantees that the document was signed by the holder of the private key corresponding to the published public key.
  • Consistency: Verification ensures that all users referencing the same version number are using identical documents.

What Verification Does Not Guarantee

  • Correctness: Verification does not guarantee that the content of the document is correct, complete, or suitable for your use case.
  • Legal Compliance: Verification does not confirm that implementing the framework satisfies any legal or regulatory requirements.
  • Security of Distribution Channels: Verification does not protect against man-in-the-middle attacks if you obtain the hash or public key from a compromised source. Always obtain verification materials from trusted, independent channels.

Best Practices

  • Always verify canonical documents before implementation or citation.
  • Obtain hashes and public keys from multiple independent sources when possible.
  • Store verified canonical documents securely to avoid re-downloading unverified copies.
  • Re-verify documents periodically if they are stored long-term.
  • Use both SHA-256 and GPG verification when both are available for maximum assurance.

Reporting Verification Failures

If verification fails:

  • Do not use the document.
  • Download a fresh copy from the official repository.
  • Verify the fresh copy using published hashes and signatures.
  • If verification continues to fail, contact the stewards to report a potential integrity issue.

Persistent verification failures may indicate a compromised distribution channel, corrupted files, or tampering.
